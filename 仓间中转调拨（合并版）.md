# 仓间中转调拨 - 物流中台核心履约能力

## 业务背景

跨境卖家在多仓履约场景下，货物需要在不同仓库之间中转调拨。由于中转链路涉及多个独立系统（ERP + 三方仓WMS + 平台仓），中间存在大量人工操作节点，效率低且容易出错。

**两大中转场景：**
- **场景A：国内三方仓→海外仓**（如发网→出口易，速卖通全托管备货、FBA备货）
- **场景B：海外仓→平台仓**（如品赢海外仓→美客多官方仓）

## 两大场景对比

| 维度 | 场景A：国内三方仓→海外仓 | 场景B：海外仓→平台仓 |
|------|------|------|
| 典型案例 | 发网→出口易、三方仓→FBA、三方仓→AE全托管备货仓 | 品赢海外仓→美客多官方仓 |
| 产品标签来源 | 接口自动获取（出口易SKU标签→上传COS→推单透传） | 客户从平台手动下载后上传ERP |
| 箱唛来源 | 接口自动获取（入库单创建成功后自动请求→落库→透传） | 客户从平台手动下载后上传ERP |
| 装箱数据来源 | 三方仓完成装箱后回传ERP | 原箱中转：客户在ERP维护；拆箱换标：海外仓回传 |
| 业务模式 | 单一模式（三方仓承担贴标装箱） | 原箱中转 / 拆箱换标 两种模式 |
| 状态机 | 无（一次性下发） | 五阶段状态机控制两阶段下发 |

### 关键接口设计

**1. 出库单下发（ERP → 发网）**
- 接口：`POST /hwcOrder/createHwcOrderApi`
- 鉴权：AES-128-ECB（companyId##loginUserId##timestamp）
- sourceId：15（速卖通仓发专用）
- SKU标签字段（#64804新增）：item数组新增commodityPDF（商品贴PDF）+ goodsLabelConfigInfo（标签配置）

**2. SKU标签来源链路**
```
① PHP 拉商品/盘点时，调出口易标签接口 → 上传COS → 存 db_hwcproduct.cos_hwcsku_label
② 判断：调拨单起始仓=发网（id=2615），目的仓=出口易（id=60）
③ 满足条件则提交 cos_hwcsku_label 给JAVA出库单下发接口
④ 支持虚拟仓判断：db_warehouse.is_virtual=1 → 找父仓 → 找 logisticsId
```

**3. 入库单下发（ERP → 出口易）**
- 创建：`POST /v1/inboundOrders`
- 取消：`GET /v1/inboundOrders/{packageId}/cancel`
- 查询：`GET /v1/inboundOrders/{shipmentId}/status`

**4. 箱唛全自动流转**
```
出口易入库单创建成功
  → JAVA 调出口易箱唛接口：GET /v1/inboundOrders/{ShipmentId}/label
  → PHP 接收箱唛：入参(requestId, order_no, print_url) → 落库 ship_batch_order.box_mark_file_path
  → PHP 调 JAVA 箱唛下发接口 → JAVA 调发网箱唛提交接口
  → 发网贴箱唛，准备出库
```

### 设计亮点

1. **出入库单联动创建**：操作出库单时勾选后自动关联生成调拨入库单，支持企业维度勾选记忆，已存在入库单时按钮置灰防重复
2. **SKU商品标签自动化**：拉商品时自动获取出口易SKU标签→上传COS落库→推单时自动透传发网，贴标作业无人化
3. **虚拟仓三层路由判断**：`is_virtual` + `parent_warehouse_id` + `logisticsId` 三层判断，精确识别发网→出口易调拨场景，不误伤其他链路
4. **装箱完成自动标记**：PHP接收装箱数据后自动判断全量完成，设置is_pack_over=1，前端展示标签，无需人工确认
5. **箱唛全自动流转**：从出口易获取箱唛→落库ERP→透传发网，全链路零人工干预
6. **报错边界设计**：「单据已下推三方仓」和「入库单未创建」两种拦截场景，保证数据一致性

### 问题根源

运营人员从美客多商家中心下载**商品标签、箱唛、预约信**三类附件，上传到ERP后统一下发给海外仓。但原有系统无法区分附件类型，仓库收到后无法判断哪个是商品标签、哪个是箱唛，导致入库效率低甚至被拒收。

### 两种业务模式

**原箱中转（transfer_biz_type=2）：**
- 国内已提前贴好平台仓商品标签
- 货物到海外仓后无需拆箱
- 客户只需通过ERP下发箱唛+预约信给海外仓
- 一次性下发，流程简单

**拆箱换标（transfer_biz_type=1）：**
- 国内未贴平台仓商品标签
- 货物到海外仓后，仓库需要拆箱→贴标→重新装箱
- 两阶段流程：
  - 阶段一：客户上传产品标签→ERP下发换标任务→海外仓拆箱贴标装箱→回传装箱数据
  - 阶段二：客户基于装箱数据从美客多获取箱唛+预约信→上传ERP→二次下发给海外仓

### 核心业务流程

**拆箱换标流程（两阶段）：**
```
阶段一（下发换标）：
  前端调PHP上传产品标签 → 保存至 file1_url
  → PHP 调 JAVA createOutboundSync 接口
  → JAVA 调海外仓【创建换标订单】API
  → 成功后 transfer_process_status = 20（按钮置灰）
  → JAVA 异步轮询海外仓获取装箱信息
  → 成功后回调PHP写入装箱数据
  → transfer_process_status = 30

阶段二（二次推送）：
  前端识别 status=30，允许上传箱唛与预约信
  → PHP 保存至 file2_url、file3_url
  → PHP 再次调 JAVA createOutboundSync
  → JAVA 调海外仓【换标更新装箱】API
  → 成功后 transfer_process_status = 50
```

**原箱中转流程（一次性）：**
```
  前端调PHP上传箱唛与预约信 → file2_url、file3_url
  → PHP 调 JAVA createOutboundSync（附带装箱数据）
  → JAVA 调海外仓【创建中转订单】API
  → 成功后 transfer_process_status = 50
```

### 五阶段状态机

| transfer_process_status | 含义 | 按钮状态 | 触发条件 |
|------------------------|------|---------|---------|
| 10 | 待下发任务 | 正常可用 | 初始状态 |
| 20 | 换标中/待装箱 | 置灰，提示「等待海外仓装箱完毕」 | 换标任务下发成功 |
| 30 | 待补全附件 | 高亮可用（上传箱唛+预约信） | 海外仓回传装箱数据 |
| 40 | 待推发货指令 | 等待系统处理 | 箱唛+预约信上传完成 |
| 50 | 任务已完成 | 完成态 | 二次下发成功 / 原箱中转下发成功 |

**按钮控制逻辑：**
- `logisticsId=3814` 且 `transfer_process_status=20`：按钮置灰，悬浮提示「当前已下发换标任务，请等待海外仓装箱完毕」
- `logisticsId=3814` 且 `transfer_process_status=30` 且 `order_status=1`：按钮高亮，可进行二次下发
