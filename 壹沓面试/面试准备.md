## 四、核心项目讲法（按壹沓场景裁剪）

### 项目 1：OpenClaw 智能采购员（最重要 —— 类比海运接单/订舱）

**为什么这个项目最关键**：
壹沓的 AI 接单、智能订舱调度，本质就是 OpenClaw 的"AI 判断 + CLI 执行"架构，**场景换一下就是同构的**。

**讲法**（2 分钟 STAR）：

> **Situation**：马帮的采购员每天要处理 400+ 单异常——跟单、找替代品、降本沟通、采购量计算，4 大流程全是高频重复操作，一个异常单要切换马帮 ERP + 1688 + 供应商微信多个系统。
>
> **Task**：把"人工逐单处理"转为"AI 自动处理 + 人工兜底"。
>
> **Action**：
> 1. **现场调研**：跟采购员跟单 2 周，把 4 大流程拆成 **5 阶段**（下单前审量 → 马帮+1688 下单 → 价格运费审批 → 催货跟单 → 降本沟通）。
> 2. **Agent 架构设计**：**五层解耦** —— 大模型 Agent + Python CLI + Go 网关 + FastAPI 业务服务 + 马帮 OpenAPI/1688 平台。核心原则是 **"AI 做异常判断，CLI 只取写数据"** —— AI 不碰数据读写，只输出决策；CLI 不做判断，只执行命令。
> 3. **最小可执行命令抽象**：以 1688 下单为例，浏览器手工操作封装为 `place-order` → `place-1688-order` 两条 CLI 命令，AI 一句话完成。同类思路覆盖催发货、审价、找替代全链路，**共 10 个 CLI 命令**独立完成产品设计 + FastAPI 后端开发。
>
> **Result**：
> - 马帮下单 + 1688 下单 + 跟单 + 审价全链路调通上线
> - 内部电商公司和品牌客户试用中
> - 规划逐步覆盖采购团队日均数千单异常自动化处理
>
> **迁移到壹沓**：这套"现场调研 → 流程拆解 → CLI 最小可执行命令 → AI 判断 + 工具执行"的方法论，**直接可以迁移到海运接单/订舱**——把采购员换成货代操作员，把 1688 API 换成船公司/港口 API，底层架构一致。

## 九、海运行业知识速成（被问到能接上话）

### 9.1 角色链条心智模型

```
发货人 Shipper (工厂/卖家)
    ↓ 委托
货代 Freight Forwarder (壹沓客户就是这批)
    ↓ 订舱
船公司 Carrier (MSC/马士基/中远海/EMC/ONE/HMM)
    ↓ 分配
码头/港口 Terminal (上海洋山/宁波北仑/深圳盐田)
    ↓ 到港
收货人 Consignee (海外买家/海外仓/FBA)
```

**壹沓智能履约本质 = 把货代内部"接单→订舱→放舱→单证"这段 SOP 自动化**。

### 9.2 海运全链路 7 阶段

| # | 阶段 | 核心动作 | 关键单据 |
|---|------|---------|---------|
| 1 | **接单** | 询价 → 报价 → 确认委托 | Booking Request / SI 草稿 |
| 2 | **订舱** | 货代向船司下 booking → 船司确认舱位 | **SO 订舱确认书** |
| 3 | **放舱** | 船司放出舱位号 → 通知工厂提箱 | SO 编号 + 提箱指令 |
| 4 | **装箱/进港** | 工厂装箱 → 进港 → VGM 申报 | VGM + 装箱单 |
| 5 | **开船/提单** | 船开 → 船司签发提单 | **MBL / HBL** |
| 6 | **到港/清关** | 到港 → 清关 → 换单 | D/O 提货单 |
| 7 | **结算** | 货代对账 → 发货人付款 | 对账单 / 发票 |

### 9.3 必背术语 10 个（脱口而出版）

| 术语 | 全称 | 一句话 |
|------|------|-------|
| **SO** | Shipping Order | 订舱确认书——船司给舱位的凭证 |
| **SI** | Shipping Instruction | 托运指令——告诉船司提单怎么写 |
| **MBL / HBL** | Master / House B/L | 主提单（船司签）/ 分提单（货代签） |
| **VGM** | Verified Gross Mass | 集装箱核实总重——SOLAS 强制申报 |
| **Cut-off** | —— | 截止时点（截货/截 VGM/截单三道） |
| **FCL / LCL** | Full / Less Container Load | 整箱 / 拼箱 |
| **FOB / CIF / DDP** | Incoterms | 贸易条款（谁负责运费/保险/清关） |
| **THC** | Terminal Handling Charge | 码头操作费 |
| **电放 / Telex Release** | —— | 船司电子指令放货，跨境电商主流 |
| **Rolling / 甩柜** | —— | 舱位被砍 → 改配下一班船 |

### 9.4 加分冷门点 3 个

- **AMS / ISF 10+2**——美线强制安全申报，错报罚金 $5000/票
- **预配（Pre-allocation）vs 确配**——旺季砍舱核心概念
- **EDI IFTMIN / IFTMBC**——大货代订舱老牌报文格式

### 9.5 三道 Cut-off 详解

| Cut-off | 含义 | 错过后果 | 能否补救 |
|---------|------|---------|---------|
| **Cargo Cut-off** | 截货——货必须进港 | **必然甩柜** | 几乎不能（闸口物理层面关闭） |
| **VGM Cut-off** | 截 VGM——重量必须申报 | 甩柜 | 可紧急补报（加急费 $50-100） |
| **SI Cut-off** | 截单——提单内容提交 | 甩柜 | 可走 Amendment（加急费 $50-200） |

**甩柜（Rolling）代价**（40HQ 美西线）：空舱费 $1500 + 改配费 $500 + 仓储费 $200/天 + 错过大促机会成本几万到几十万。

**甩柜 3 大原因**：
- **船司 Overbooking 超订舱**：50%（旺季尤甚，Rolling Reason Code R01）
- **客户货未就绪**：30%（R02，工厂延误 / 拖车延误）
- **不可抗力**：20%（R04，天气 / 船坏）

**甩柜后标准 SOP 6 步**（全部是 AI Agent + CLI 的切入点）：通知客户 → 重新订舱 → 改签单据 → 费用核算 → 海关处理 → 赔偿谈判。

### 9.6 单点 RPA vs 流程 RPA vs AI Agent

| 类型 | 通俗讲 | 海运例子 |
|------|--------|---------|
| **单点 RPA** | 把**一个孤立动作**交给机器人 | 每天登录船司官网下载船期表 |
| **流程 RPA** | 把**一段连续 SOP** 串成机器人跑 | 登录 → 查舱位 → 填订舱单 → 提交 → 邮件回传 |
| **AI Agent** | 有**判断能力**，能处理异常和分支 | 订舱被砍？自动切备选船司 + 改航线 + 通知客户 |

**单点 RPA 三个硬伤**：脚本死板（页面改版就废）/ 不会判断（异常全靠人工兜底）/ 跨系统断链（连不起多系统流程）。

**壹沓差异化**：**底层有 RPA 执行器（干脏活）+ 上层有 AI Agent（做判断）** = 对应你 OpenClaw 的 **"AI 判断 + CLI 执行"五层解耦**。

## 十、海运 AI 切入点全景（7 阶段 × 你的项目映射）

### 切入点 ① 接单：AI 单证识别 + 智能报价

**痛点**：客户发来的 Booking 千奇百怪（邮件 / PDF / Excel / 微信截图），操作员手工录入一票 8-15 分钟，品名口语化严重（"电视机"要映射到 `LCD Monitor 8528.72`）。

**映射你的项目**：**完全同构于智能优选渠道**——"物流商"换成"船司"；多维测算权重（运费/时效/稳定性）+ 客户自定义权重 = 直接复用；**Y2 多尾程组合优化的"分段 top N → 笛卡尔积 → 全链路重测"逻辑**搬到"头程+中转+尾程船司组合"完全对齐。

### 9.6 「单点 RPA」vs「流程 RPA」vs「AI Agent」

面试中这个词出现概率极高（壹沓 RPA 起家），务必能区分：

| 类型 | 通俗讲 | 例子 |
|------|--------|------|
| **单点 RPA** | 自动化**一个孤立动作**，无判断无编排 | 每天登录船司官网下载船期表存本地 |
| **流程 RPA** | 把**一段连续 SOP** 串起来跑 | 登录船司 → 查舱位 → 填订舱单 → 提交 → 截图回邮件 |
| **AI Agent** | 有**判断能力**，能处理异常和分支 | 订舱被砍？自动切到备选船司 + 改航线 + 通知客户 |

**单点 RPA 三大硬伤**：
1. 脚本死板——船司页面改版一次脚本全废
2. 不会判断——异常全靠人工兜底
3. 跨系统断链——连不起船司官网 + EDI + ERP + 邮件 + 微信

**壹沓的打法**：底层 RPA 执行器（干脏活）+ 上层 AI Agent（做判断）= **「执行底盘 + 判断大脑」双层架构**。

**和 OpenClaw 同构**：
| 壹沓 | OpenClaw |
|------|----------|
| AI Agent 判断层 | 大模型 Agent 层 |
| RPA 执行器 | Python CLI 层 |
| 业务服务 | FastAPI 业务层 |
| 外部系统 | 马帮 / 1688 OpenAPI |

## 十、海运 AI 切入点全景（★ 这是面试最能出差异化的部分）

### 切入点 ① 接单：AI 单证识别 + 智能报价

**痛点**：客户 Booking 格式千奇百怪（邮件 / PDF / Excel / 微信截图 / 传真）→ 操作员手工录入 8-15 分钟/票。

**AI 做什么**：
- 多模态识别（GPT-4V / Qwen-VL）直接吃图/PDF → 结构化字段
- RAG 查历史订单 + HS Code 知识库自动映射
- 置信度分流（>90% 直出 / 中间段人工复核 / <60% 退回客户）
- 多船司并行比价生成报价单

**对标你项目**：**Echo RAG + Dify 工作流** = 技术底盘；**OpenClaw badcase 驱动** = 持续迭代方法。

**ROI**：8-15 分钟 → 30 秒；日处理 30 票 → 150 票。

---

### 切入点 ② 订舱：智能订舱决策引擎（最核心）

**痛点**：货代盯多船司官网 / EDI / INTTRA 比价，老操作员靠经验选船司；旺季砍舱频繁；新人玩不转。

**AI 做什么**：
1. **多船司并行比价**：Agent 同时调 MSC / 马士基 / COSCO / CMA API 或 RPA
2. **砍舱概率预测**：基于历史 Rolling Reason Code 训练——"航线+时段+船司"砍舱概率 X%
3. **组合订舱优化**：5 票货拆 2+3 两家 vs 一家整发，算总成本最优
4. **旺季多船司占位**：2-3 家预配并行，哪家确配用哪家

**对标你项目**：**完全同构于智能优选渠道**——物流商换成船司，逻辑一模一样；**Y2 多尾程分段 top N + 笛卡尔积 + 全链路重测**的逻辑搬到"头程+中转+尾程船司组合"完全对齐。

**ROI**：订舱 30 分钟/票 → 3 分钟；运费下降 5-15%；砍舱率 3-5% → <1%。

---

### 切入点 ③ 放舱：SO 自动解析 + 拖车调度

**痛点**：船司给 SO 后要解析 PDF 拿船名/航次/舱位号、通知工厂/客户、同步录 TMS、排拖车档期。

**AI 做什么**：
- Vision 模型识别 SO PDF + 规则校验
- 按客户习惯生成 WeChat / 邮件 / 短信通知
- 结合提箱时间 + 距离 + 拖车档期自动调度

**对标你项目**：**OpenClaw "AI 判断 + CLI 执行"架构直接对应**——AI 解析 SO 有效性，CLI 调拖车调度 API。

**ROI**：通知 15 分钟 → 30 秒；漏通知率 2-3% → 0。

### 切入点 ④ 装箱/进港：Cut-off 预警 + 单证自动化

**痛点**：三道 Cut-off 每票要人盯，漏一个甩柜；VGM 漏报上不了船；各国单证格式不同（AMS / ENS / ISF 10+2）。

2. **VGM 自动申报**：对接工厂地磅 / 客户 ERP → 称重自动推船司
3. **单证批量生成**：Booking → 装箱单 / 发票 / 报关单 / AMS 申报
4. **敏感商品预检**：HS Code + 目的国 → OFAC 制裁 / 出口管制

**对标你项目**：**Y2 下 Call 7 档时点（事件触发+延时任务+外部时点）= 直接原型**；**压缩面单批量生成方法论 = 直接复用**。

> **一是 Y2 货代全链路我是产品负责人**——2024 年关税战后，很多 Temu 卖家从直发切到 Y2 货代模式：**国内揽收 → 头程 → 海外仓 → 尾程派送**。我主导了多尾程渠道选择、预约揽收时点下发、下 Call 时机 7 档枚举整套能力。**虽然是空运货代不是海运，但货代这件事的产品语言——承运商、舱位、时效、报关、单证——底层是通的**。
>
> **二是 1800+ 物流商对接的标准化方法论**——不管是船公司、港口代理、报关行还是货代公司，本质都是"多异构接口 → 统一中台 → 可规模化接入"。**我这套 sourceId 路由 + 状态机约束 + 异步推送框架，换到海运场景同样能跑**。
>
> 海运特有的放舱规则、舱位预测、BL/SWB 单证格式我需要 **2-3 周时间补齐**，但 **产品方法论和业务敏感度不需要从 0 学**。"

**加分动作**：面试前 **花 1 小时查 3 个术语**——SO（订舱委托）、MBL/HBL（主单/分单）、VGM（验证箱重）。面试中主动用一次，瞬间显得"虽然没做过但做过功课"。

### 追问 2：**"带过团队吗？你带几个人？"**（JD 要求产品团队搭建，灰色）

**原则**：**诚实承认正式下属少 + 用协同 Owner 经验 + 方法论化反打**，不能吹。

**标准答**：

> "**正式直接下属我只带过 1-2 个**，但 **产品线协同 Owner 经验是主战场**：
>
> - **马帮物流中台开放平台** 这条线，我是产品负责人，**同时协同 3 个研发小组**（JAVA 中台组、PHP ERP 组、前端组，合计 10 余人），跨组的需求评审、排期、交付节奏全在我这边过
> - **OpenClaw 项目**，我是独立全栈 Owner，产品设计 + FastAPI 后端开发一个人扛，**和大模型 Agent 工程师、采购业务方三方协同**推进
> - **头部客户定制线**，和销售、售前、交付、技术 4 个角色常态化跨部门拉齐，完美日记这种百万级项目就是典型的跨部门协同闭环
>
> **做管理我有三件事想得比较清楚**：
> 一是 **招人标准**——产品岗招'能自己把一块业务跑通闭环'的人，不招只做需求传话筒的；
> 二是 **考核锚点**——B 端产品不看需求数量，看 **用户行为指标 + 客户续费 / 定制营收 + 协同评分** 三维综合；
> 三是 **梯队成长**——老人带新人模式 + 每人 Own 一条子产品线 + 季度复盘迭代方法论。
>
> **如果贵司这个岗位需要从 0 搭建海运产品团队**，我愿意坦诚：这是 **我职业生涯第一次做正式的团队管理**，但我做过的协同 Owner 工作量和挑战度，基本已经等于一个小团队负责人了。"

**坦诚承认**是这个问题的核心——面试官比起"吹牛说带过 10 人"，更吃"承认短板 + 说清楚我怎么补"。
